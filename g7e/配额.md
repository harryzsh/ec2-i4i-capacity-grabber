# 配额（抢 g7e.48xlarge 前必读）

抢 g7e.48xlarge 之前，**先把配额提够**，否则就算抢到 ODCR 也起不了实例。

## 提哪个配额

g7e 属于 EC2 **G 系列**，要提的是：

| 项 | 值 |
|----|----|
| 配额名 | **Running On-Demand G and VT instances** |
| Quota code | `L-DB2E81BA` |
| Service | `ec2`（Amazon EC2） |
| 单位 | **vCPU**（不是台数！） |
| 每台 g7e.48xlarge | **192 vCPU** |

**换算**：要抢 N 台 → 配额至少 **192 × N** vCPU。

| 想抢台数 | 至少需要 vCPU |
|----------|---------------|
| 1 台 | 192 |
| 2 台 | 384 |
| 4 台 | 768 |
| 8 台 | 1536 |

> ⚠️ 配额是**分 Region** 的。你在哪个 region 抢，就在哪个 region 提。

## 方式一：Console 提（最简单）

1. 登录后打开 **Service Quotas** 控制台：`https://console.aws.amazon.com/servicequotas/`
2. 左侧菜单 **AWS services** → 搜索 **EC2** → 点 **Amazon Elastic Compute Cloud (Amazon EC2)**
3. **右上角先把 Region 切到目标区**（如 `us-east-1`）——配额分区，切错区白提
4. 在配额搜索框输入 **`Running On-Demand G and VT instances`** → 点进该配额
5. 右上角点 **Request increase at account level**（账户级申请）
6. 在 **Increase quota value** 里填目标 **vCPU 数**（注意是 vCPU！要 N 台就填 `192 × N`，如 4 台填 `768`）→ 点 **Request**
7. 跟踪审批：左侧 **Request history** 查看状态。大额 GPU 申请常转人工审核，急的话再去 **Support → Create case → Service limit increase** 补一个 case 说明用途催进度

> 小幅提升常自动批；一次性要很大量（尤其稀缺 GPU）通常转人工，可能要几小时到几天。

## 方式二：CLI 提

```bash
# 1. 查当前 G/VT vCPU 配额
aws service-quotas get-service-quota \
  --service-code ec2 --quota-code L-DB2E81BA --region us-east-1

# 2. 申请提到目标 vCPU（示例：4 台 = 768）
aws service-quotas request-service-quota-increase \
  --service-code ec2 --quota-code L-DB2E81BA \
  --desired-value 768 --region us-east-1

# 3. 查看申请历史 / 状态
aws service-quotas list-requested-service-quota-change-history-by-quota \
  --service-code ec2 --quota-code L-DB2E81BA --region us-east-1
```

## 方式三：直接用脚本预检（推荐抢前跑一次）

脚本自带 `--check-quota`，读 Service Quotas 当前配额，并对照你的目标台数判断够不够：

```bash
# 检查当前 region 的 G/VT 配额，对照目标台数
python3 grab_g7e_odcr.py --check-quota --azs us-east-1b us-east-1d --per-az-count 2

# 输出示例：
#   G/VT On-Demand quota (Running On-Demand G and VT instances, L-DB2E81BA) in us-east-1: 768 vCPU
#     -> allows up to 4 x g7e.48xlarge (192 vCPU each)
#     target 4 instance(s) needs 768 vCPU -> OK
```

`--check-quota` 只读、不抢、不计费。它额外需要一个 IAM 权限：`servicequotas:GetServiceQuota`（见 README 的 IAM 说明）。

## 两个一定要记住的事

1. **配额按 vCPU，不按台数**——填 `768` 不是填 `4`。
2. **配额批了 ≠ 抢得到容量**。配额只决定“允许你起多少”，真正能不能起还要看那个 AZ 有没有货——所以才要 `grab_g7e_odcr.py --watch` 死等 ODCR。
