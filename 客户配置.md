# 客户侧配置：让 ASG 自动吃掉抢到的预留

脚本（`grab_odcr.py`）只负责**把 open 预留抢到手**。预留能不能被客户的 Auto Scaling Group（ASG）自动「吃掉」、扩容时实例真正落进预留，**取决于客户侧的 ASG 配置**。本文是开 ASG 前客户必须确认的事项。

> 脚本对客户的 ASG / 启动模板 / 节点**一律不读、不改、不删**，只调 3 个 ODCR API（建/查/取消预留）。两边各管各的、互不干涉。

---

## 一、核心开关：`capacity-reservations-first`

在 **ASG 上**（不是启动模板上）设置 Capacity Reservation 偏好：

```bash
# 新建 ASG：
aws autoscaling create-auto-scaling-group \
  --auto-scaling-group-name <你的ASG名> \
  --launch-template "LaunchTemplateId=<LT-ID>,Version=\$Latest" \
  --vpc-zone-identifier "<1b子网>,<1d子网>" \
  --capacity-reservation-specification "CapacityReservationPreference=capacity-reservations-first"

# 已有 ASG 改配置：
aws autoscaling update-auto-scaling-group \
  --auto-scaling-group-name <你的ASG名> \
  --capacity-reservation-specification "CapacityReservationPreference=capacity-reservations-first"
```

| 偏好值 | 行为 | 是否推荐 |
|--------|------|------|
| `capacity-reservations-first` | 优先用匹配的 open 预留，预留用光了**自动回落到普通 On-Demand**（软兜底，不会扩容失败） | ⭐ **推荐** |
| `capacity-reservations-only` | 只用预留，没预留就**扩容失败** | ❌ 太硬 |
| `none` / `default` | 不主动用预留 | ❌ 不适用 |

> 用 `capacity-reservations-first` + `open` 预留，ASG **不需要**指定预留 ID，属性匹配的实例自动落入——这就是脚本默认建 `open` 预留的原因，客户 ASG 零改动即可吸纳。
>
> ⚠️ 回落的普通 On-Demand 实例**占用同一个 `L-1216C47A` vCPU 配额**。所以配额要在目标核数之上留 20% 余量（9,984 → 12,000），否则预留占满后回落实例会撞配额起不来。详见 [`扩大.md`](扩大.md)。

---

## 二、实例要落进预留，4 个属性必须 100% 对齐

少一项对不上，实例就**静默走普通 On-Demand**（不报错），结果**双倍计费**：空转预留 + On-Demand 各付一份。

| 属性 | 必须一致 | 本次配置 |
|------|----------|----------|
| 机型 instance type | 预留机型 = 启动模板机型 | `i4i.16xlarge` |
| 平台 platform (OS) | 预留 platform = 实例 OS | `Linux/UNIX`（AL2023） |
| 可用区 AZ | 预留 AZ ∈ ASG 子网所在 AZ | `us-east-1b` / `us-east-1d` |
| 租户 tenancy | 预留 tenancy = 实例 tenancy | `default`（共享） |

> **匹配只看这 4 个属性，`EbsOptimized` 不在其中。** 脚本建预留时显式设了 `EbsOptimized=True`（因为 i4i 是 Nitro 机型、EBS 优化强制开启关不掉），但这只是让预留属性诚实对齐，**不参与 open 匹配、也不会因此卡住实例落入**。客户启动模板无论写不写 `EbsOptimized`，i4i 实例起来都是 EBS 优化的，天然一致，不用管。

---

## 三、常见坑

- **属性对不上 = 双倍计费**：见上表，开 ASG 前务必核对 4 个属性。
- **AMI 架构不匹配**：i4i 是 Intel x86_64，启动模板别配 ARM/Graviton 的 AMI，否则实例根本起不来。
- **预留用了 `targeted` 而非 `open`**：targeted 要求实例显式指定预留 ID，ASG 不会自动吸纳。脚本默认 `open`，别改。
- **ASG 50/50 调度 vs 预留分布**：ASG 默认往实例数少的 AZ 放，最终≈50/50。所以预留也要 50/50 均衡铺（脚本用 `--per-az-cores`），否则多的那边空转、少的那边缺口落到 On-Demand。

## 四、EKS 自建节点组专项

> 针对客户实际架构：**EKS 自建节点组 + 自管 auto scaler（非 K8s Cluster Autoscaler）+ 每个 AZ 一个 ASG**。
> 这套架构天然规避了两个常见坑，但引入一个新风险。

### 已规避的两个坑 ✅

| 坑 | 为什么不成立 |
|------|----------------------|
| **AZRebalance**（单 ASG 跨多 AZ 会搬实例丢容量）| 只在单 ASG 横跨多 AZ 时触发。客户每 AZ 一个 ASG（单 AZ），组内无可均衡对象 → 不触发。前提：每个 ASG 只挂单 AZ 子网 |
| **跨 AZ 单节点组**（无法精确控制 AZ 落点）| 客户已是每 AZ 一个 ASG，正好对齐脚本 `--per-az-cores` 的均衡设计 |

### 🔴 真正剩下的核心坑：自管 scaler 低负载缩容

抢到 i4i 不等于守得住。客户自管 scaler 若有「利用率低 → 缩容」逻辑，会在 pod 没填满时主动终止节点，把容量还回 AWS，高峰**可能抢不回**。

**防护（推荐都做）**：

1. **大促窗口把每个 ASG 的 `min = desired = 该 AZ 预留数` 钉死**，关掉向下缩容——最干净，坑直接消除。
2. **大促前主动预热**：拉高 `min/desired` 强制起节点占满预留，别等 scaler，否则 ODCR 空转全额计费。

```bash
# 钉死下限（每个单 AZ ASG 各做一次，N = 该 AZ 预留数）
aws autoscaling update-auto-scaling-group \
  --auto-scaling-group-name <该AZ的ASG名> \
  --min-size <N> --desired-capacity <N> --max-size <N>
```

> ✅ **这是唯一需要跟客户在 K8s 层敲死的事**：大促期间把两个 ASG 的 `min/desired` 钉到预留数、关掉向下缩容。

### 🟡 两个「配置不对就用不上预留」的坑

| 坑 | 正确做法 |
|----|----------|
| 启动模板配多机型 → open 预留四要素匹配不上，静默走 On-Demand | 启动模板**只配 `i4i.16xlarge`** |
| 大促期间滚动更新 / 改 LT 版本 → recycle 节点可能丢容量 | 配置**冻结**到大促结束，禁止改 LT、禁止滚动更新 |

## 五、抢货前 checklist

**EC2 层硬阻塞**（错一个整个方案归零，最优先）：

- [ ] **抢预留的账号 = EKS 集群账号**（open 预留只在同账号内匹配）
- [ ] **该账号 On-Demand Standard vCPU 配额 ≥ 12000**（156 台 = 9984 vCPU + 回落余量；默认配额远不够 → 提早开工单，**头号硬阻塞**，详见 [`扩大.md`](扩大.md)）
- [ ] **两个 ASG 的子网确实在 1b + 1d**
- [ ] 启动模板只锁 `i4i.16xlarge`、`Linux/UNIX`、`default` 租户、x86_64 AMI

**客户侧配置**：

- [ ] 每个 ASG 设了 `CapacityReservationPreference=capacity-reservations-first`
- [ ] 每个 ASG 只挂**单 AZ 子网**（可顺手 `suspend-processes AZRebalance` 防漂移，零成本）
- [ ] **大促窗口把每个 ASG `min/desired` 钉到该 AZ 预留数，关掉向下缩容**
- [ ] 实例打了 `kubernetes.io/cluster/<集群名>: owned` 标签（否则节点不入集群）
- [ ] 关键 pod 配了 PodDisruptionBudget
- [ ] 配置已冻结，大促期间不改 LT、不滚动更新

> **一句话**：客户「自管 scaler + 每 AZ 一个 ASG」已规避 AZRebalance 和跨 AZ 调度两个坑；K8s 层只剩一件事——**钉死 `min/desired`、别向下缩容**。真正的拦路虎在 EC2 层：**账号对齐 + vCPU 配额**。
